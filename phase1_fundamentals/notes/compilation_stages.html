Section 01

Basic Input & Output in C++

C++ provides two families for I/O: the C++ stream library (<iostream>, etc.) and the inherited C stdio library (<cstdio>). You almost always want the C++ streams; <cstdio> is there when you need raw performance or interop with C.

Headers and What They Provide

HeaderKey Objects / FunctionsPurpose
<iostream>cin, cout, cerr, clogBuffered & unbuffered stream I/O
<fstream>ifstream, ofstream, fstreamFile I/O
<sstream>istringstream, ostringstreamIn-memory string I/O
<iomanip>setw, setprecision, fixed, hex…Stream formatters
<cstdio>printf, scanf, fprintf, fgets…C-style I/O (faster, less safe)

std::cout — Standard Output

cout writes to the terminal. Output is buffered: data accumulates in a buffer and is flushed all at once. Use << (insertion operator) to send data.

#include <iostream>
#include <iomanip>
using namespace std;

int main() {
    int    age   = 21;
    double grade = 9.75;
    cout << "Name  : Arjun\n";
    cout << "Age   : " << age << "\n";
    cout << "GPA   : " << fixed << setprecision(2) << grade << "\n";
    cout << "Loading..." << flush;  // flush manually when needed
    return 0;
}
Note Prefer "\n" over endl inside loops. endl flushes the buffer every call — 10–50× slower in tight loops.

std::cin — Standard Input

cin reads from the terminal using >> (extraction operator). For full lines use getline(cin, str).

#include <iostream>
#include <string>
using namespace std;

int main() {
    int n;
    string name;
    cout << "Enter count: ";
    cin  >> n;
    cin.ignore();        // discard leftover '\n'
    cout << "Enter name: ";
    getline(cin, name);
    cout << name << " x " << n << "\n";
}
Pitfall Mixing >> and getline without cin.ignore() causes getline to immediately return empty — it consumes only the leftover '\n'.

std::clog — Buffered Diagnostic Log

clog writes to stderr but is buffered (unlike cerr). Use it for high-volume diagnostic output during development where separation from cout is needed but immediate flushing is not.

int processFile(const string& path) {
    clog << "[LOG] Processing: " << path << "\n";
    // ... actual work ...
    clog << "[LOG] Done\n";
}
When to use which
• cout — normal program output users see.
• cerr — error messages that must appear immediately (unbuffered).
• clog — internal diagnostics / debug tracing (buffered, goes to stderr).

<cstdio> — C-style I/O

When you need maximum throughput or are calling C APIs, <cstdio> provides printf and scanf — faster but not type-safe.

#include <cstdio>

int main() {
    int   marks   = 95;
    float percent = 95.5f;
    printf("Marks: %d  Percent: %.2f\n", marks, percent);

    char buf[64];
    scanf("%63s", buf);  // always limit width
    fprintf(stderr, "Got: %s\n", buf);
}
Performance tip If mixing <cstdio> and <iostream>, call ios_base::sync_with_stdio(false) and cin.tie(nullptr) at the start of main to unlock C++ stream speed — but don't mix the two families after that.
Section 02

Compilation Stages of C++

When you "compile" a C++ file, the toolchain secretly runs four distinct passes, each transforming the source into a lower-level representation.

STAGE 1
Preprocessing
.cpp → .i
STAGE 2
Compilation
.i → .s
STAGE 3
Assembly
.s → .o
STAGE 4
Linking
.o → ELF/exe

Stage 1 — Preprocessing (cpp)

The C Preprocessor (cpp) runs before the compiler sees any code. It is a pure text transformation engine.

What it does:

  • Expands #include — literally pastes header file content in place.
  • Expands #define macros by text substitution.
  • Evaluates #if / #ifdef / #ifndef / #endif conditionals.
  • Removes all comments.
  • Inserts #line directives to map errors back to original files.
# Stop after preprocessing — inspect the .i file
g++ -E main.cpp -o main.i

# See macros defined for current target
g++ -dM -E -x c++ /dev/null

# main.i is typically 20,000+ lines even for a simple program
wc -l main.i

Hands-on: conditional compilation

#include <iostream>
#define DEBUG_MODE 1

#ifdef DEBUG_MODE
  #define LOG(x) std::clog << "[DBG] " << (x) << "\n"
#else
  #define LOG(x)   // expands to nothing in release builds
#endif

int main() {
    LOG("Program started");
    // Release build: LOG() vanishes completely — zero overhead
}
Troubleshoot If you get "identifier not found" — the #include path is probably wrong. Run with -E and inspect the top of the .i file, or add -v to see include search paths.

Stage 2 — Compilation (cc1plus)

The actual compiler takes the preprocessed .i file and produces assembly code (.s). This is where syntax/type checking, template instantiation, and optimizations happen.

# Stop after compilation — inspect assembly
g++ -S main.cpp -o main.s

# With optimizations — compare output
g++ -S -O2 main.cpp -o main_opt.s

# Interleave C++ source with assembly for readability
g++ -S -O0 -fverbose-asm main.cpp -o main_verbose.s

Seeing the optimization difference

int square(int x) { return x * x; }
int main()         { return square(5); }
// -O0: calls the function. -O2: collapses to "mov eax, 25 ; ret"
Troubleshoot Template errors explode with pages of mangled names. Use c++filt to demangle: g++ ... 2>&1 | c++filt. Or use -ftemplate-backtrace-limit=0.

Stage 3 — Assembly (as)

The GNU Assembler (as) takes the .s file and produces a binary object file (.o) in ELF format. The object file contains machine code but with unresolved symbol references left as placeholders.

# Assemble only
g++ -c main.cpp -o main.o

# Inspect the object file
nm -C main.o        # symbol table (-C demangles names)
objdump -d main.o   # disassemble
readelf -h main.o   # ELF header

In nm output: T = defined in text section, U = undefined (resolved later), D/B = data/BSS.

Stage 4 — Linking (ld)

The linker takes all .o files plus libraries (.a static, .so dynamic) and resolves all symbol references, producing the final executable.

# Compile and link in one step
g++ main.cpp utils.cpp -o program

# Compile separately then link
g++ -c main.cpp   -o main.o
g++ -c utils.cpp  -o utils.o
g++ main.o utils.o -o program

# Link against libraries
g++ main.o -lm -o program        # math library
g++ main.o -lpthread -o program  # pthread
ldd program                       # see runtime dependencies
Common linker errors
• undefined reference to 'foo' — declared but never defined, or forgot to link the file.
• multiple definition of 'bar' — defined in two .cpp files; put it in one, or mark inline.
• cannot find -lxyz — install dev package or add -L/path/to/lib.

Complete hands-on walkthrough

// demo.cpp
#include <iostream>
int main() { std::cout << "Hello, pipeline!\n"; }

--- Terminal ---
g++ -E demo.cpp -o demo.i   # 1. preprocess
g++ -S demo.i   -o demo.s   # 2. compile
g++ -c demo.s   -o demo.o   # 3. assemble
g++    demo.o   -o demo     # 4. link
./demo
Section 03

Macros — Power, Pitfalls & Modern Alternatives

A macro is a directive processed by the preprocessor — pure text substitution before the compiler runs. The compiler never sees a macro; it only sees the expanded result.

Object-like Macros (Constants)

#define PI         3.14159265
#define MAX_USERS  1000
#define GREETING   "Vanakkam"

int main() {
    double area = PI * 5.0 * 5.0;
    // after preprocessing: double area = 3.14159265 * 5.0 * 5.0;
}
Prefer constexpr — gives type safety, scoping, and debugger visibility that #define cannot provide.

Function-like Macros

Arguments are substituted as text, not evaluated first — this causes the famous double-evaluation bug.

#define SQUARE(x)  ((x) * (x))
#define MAX(a, b)  ((a) > (b) ? (a) : (b))

int i = 3;
int r = SQUARE(i++);
// expands to: ((i++) * (i++))  — i incremented TWICE — undefined behaviour!

The parenthesis rule — always wrap args and result

// Wrong — operator precedence bug
#define DOUBLE(x)  x + x
int r = 10 * DOUBLE(3);   // 10 * 3 + 3 = 33, not 60!

// Correct
#define DOUBLE(x)  ((x) + (x))
int r = 10 * DOUBLE(3);   // 10 * (3 + 3) = 60 ✓

Variadic Macros & Stringification

// # turns argument into a string literal
#define TOSTR(x)     #x
cout << TOSTR(3.14);   // prints: 3.14

// ## concatenates tokens
#define CONCAT(a, b) a##b
int CONCAT(my, Var) = 42;  // int myVar = 42;

// __VA_ARGS__ for variadic macros
#define LOG(fmt, ...) fprintf(stderr, fmt"\n", __VA_ARGS__)
LOG("Score: %d at level %d", score, level);

Where Macros Are Genuinely Useful

  • Include guards — preventing double-inclusion of headers.
  • Conditional compilation — platform-specific code, debug/release builds.
  • Stringification & token pasting — impossible with templates.
  • Cross-platform shims — e.g. NOMINMAX on Windows.
  • X-macro pattern — generate repetitive code from a single table.
// Include guard pattern
#ifndef UTILS_HPP
#define UTILS_HPP
// ... header contents ...
#endif

// Or simply (all modern compilers):
#pragma once

Where Macros Are Dangerous

Dangerous macro
#define MIN(a,b) a<b?a:b

int r = MIN(2+3, 4);
// expands to: 2+3<4?2+3:4
// = 2+(3<4?2+3:4) = 5  WRONG!
Safe alternative
template<typename T>
constexpr T min_val(T a, T b) {
    return a < b ? a : b;
}
// Type-safe, debuggable,
// no double-evaluation.
ProblemExampleFix
Double evaluationMAX(i++, j)inline / template function
Operator precedenceDOUBLE(2+3)*4Always wrap in (( ))
No type safetyMAX("a", 2)template with concept
No scope#define SIZE 10 pollutes globallyconstexpr / enum class

Modern C++ Replacements

// Instead of: #define PI 3.14159
constexpr double PI = 3.14159265358979;

// Instead of: #define SQUARE(x) ((x)*(x))
template<typename T>
constexpr T square(T x) { return x * x; }

// Instead of: #define MAX_USERS 1000
enum class Limits { MaxUsers = 1000, MaxGroups = 50 };

// Instead of header guards:
#pragma once
Section 04

Memory Layout in Compilation Stages

When the linker produces an executable, it arranges data into distinct segments. At runtime the OS loads these into virtual memory in a fixed order.

The Virtual Address Space (Linux, 64-bit)

High addr
Kernel space
OS — not accessible
Stack ↓
local vars, call frames
· · · · · (unmapped gap) · · · · ·
Heap ↑
new / malloc
BSS segment
uninit globals, zero-filled
Data segment
init globals & statics
Low addr
Text segment
machine code, read-only

Each Segment Explained

Text Segment (Code Segment)

Contains compiled machine instructions. Read-only and shared — multiple processes running the same program share this page in physical RAM. Writing to it causes a segfault.

// All function bodies live here after compilation
int add(int a, int b) { return a + b; }
// The machine code for 'add' is in .text section

# Inspect with:
objdump -d -M intel program | head -40
size program  # shows text/data/bss sizes

Data Segment (.data)

Holds initialized global and static variables. Initial values are stored in the executable file and loaded at startup.

int    globalCount  = 42;         // .data — initialized global
double convRate     = 83.5;       // .data
const char* label   = "CGPA";    // pointer in .data; string literal in .rodata

static int callCount() {
    static int n = 0;  // .data — persists across calls
    return ++n;
}

BSS Segment (.bss)

Block Started by Symbol. Uninitialized (or zero-initialized) globals. Takes zero bytes on disk — just a size; OS zero-fills in memory. Keeps executables small.

int    score;            // .bss — default-initialized to 0
double buffer[10000];  // .bss — 80KB in memory, 0 bytes on disk!

# Proof:
g++ -c bss_demo.cpp -o bss.o
size bss.o       # large bss, tiny data
ls -lh bss.o     # file is small despite huge array

Heap

The free store — dynamically allocated memory via new / delete. Grows upward. Fragmentation and leaks happen here.

int* arr = new int[100];
delete[] arr;         // must free or memory leaks

// Modern C++ — smart pointers (no manual delete)
#include <memory>
auto ptr = std::make_unique<int[100]>();  // auto-freed at scope exit

# Detect leaks:
valgrind --leak-check=full ./program

Stack

Grows downward. Each function call pushes a stack frame containing return address, saved registers, local variables, and arguments. Extremely fast; limited to ~8 MB.

void foo(int x) {
    int  local = 10;  // stack — frame of foo()
    char buf[256];   // stack — 256 bytes in foo's frame
    // when foo() returns, frame is popped instantly
}

// Stack overflow:
void infinite() { infinite(); }  // segfault — stack exhausted

# Check stack size:
ulimit -s           # default 8192 KB
ulimit -s unlimited  # increase (current shell only)

How Variables Map to Segments

#include <iostream>

int       g_init   = 5;        // .data
int       g_uninit;             // .bss
const int LIMIT    = 100;     // .rodata

void show() {
    static int count = 0;   // .data (static local)
    int  local = 42;         // stack
    int* ptr   = new int(7); // ptr on stack, *ptr on heap

    std::cout
        << "global_init  addr: " << &g_init   << "\n"
        << "global_uninit addr: " << &g_uninit << "\n"
        << "local        addr: " << &local    << "\n"
        << "heap         addr: " << ptr       << "\n";
    // Run this — you'll see 3 distinct address ranges!
    delete ptr;
}
int main() { show(); }

Hands-on: inspect segments
• size program — shows .text / .data / .bss sizes.
• readelf -S program — all ELF sections with addresses.
• objdump -x program — full header + symbol table.
• cat /proc/<PID>/maps — live memory map of a running process.
